## ‚õëÔ∏è **Lambda Execution Roles** _(what your function can access)_

Every AWS Lambda function requires an **IAM execution role** that defines what AWS services the function can interact with. Without the right permissions, your Lambda function **cannot access** resources like S3, DynamoDB, or CloudWatch.

### üîë **What is an Execution Role?**

- An **IAM role** assigned to the Lambda function at creation.
- It contains **policies** that allow access to specific AWS services.
- Without the correct role, Lambda **cannot function properly**.

---

### üéØ **Example: Basic Lambda Execution Role**

By default, AWS Lambda needs permission to write logs to **Amazon CloudWatch** for monitoring. Here‚Äôs an **IAM role policy** that allows Lambda to write logs:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["logs:CreateLogGroup", "logs:CreateLogStream", "logs:PutLogEvents"],
      "Resource": "arn:aws:logs:*:*:*"
    }
  ]
}
```

> **This policy allows Lambda to:**
>
> - Create new CloudWatch Log Groups & Streams
> - Write logs for debugging and monitoring

---

### üì¶ **Example: Lambda Role for S3 Access**

If your Lambda function needs to read from or write to an **S3 bucket**, you must add **S3 permissions** to the IAM role.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

> **This policy allows Lambda to:**
>
> - **Read (`GetObject`) and write (`PutObject`) files** inside `my-bucket`
>
> üí° **Best Practice:** Always follow **the principle of least privilege**‚Äîgive Lambda only the permissions it absolutely needs!

### üõ†Ô∏è **How to Assign an Execution Role to a Lambda Function**

#### **üñ±Ô∏è Using AWS Console**

- **1Ô∏è‚É£** Go to **AWS Lambda Console**
- **2Ô∏è‚É£** Select your **Lambda function**
- **3Ô∏è‚É£** Navigate to **Configuration** ‚Üí **Permissions**
- **4Ô∏è‚É£** Click **"Edit" Execution Role**

#### **üßë‚Äçüíª Using AWS CLI**

**1Ô∏è‚É£ Create an IAM Role for Lambda:**

```sh

--assume-role-policy-document file://trust-policy.json
```

> This command creates a new IAM role called `LambdaExecutionRole` with the trust policy defined in `trust-policy.json`.

üëâ **Example trust policy (`trust-policy.json`):**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

**2Ô∏è‚É£ Attach an IAM Policy (e.g., CloudWatch Logs & S3 Access):**

```sh
aws iam attach-role-policy --role-name LambdaExecutionRole \
--policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
```

```sh
aws iam put-role-policy --role-name LambdaExecutionRole \
--policy-name S3AccessPolicy \
--policy-document file://s3-policy.json
```

> The first command attaches basic Lambda logging permissions.
> The second command adds S3 permissions using `s3-policy.json` (defined earlier).

**3Ô∏è‚É£ Assign the Role to a Lambda Function:**

```sh
aws lambda update-function-configuration --function-name myLambdaFunction \
--role arn:aws:iam::123456789012:role/LambdaExecutionRole
```

> Replace `myLambdaFunction` with your actual function name and `123456789012` with your AWS account ID.

## üìû **Lambda Resource Policies** _(who can invoke it)_

> While **execution roles** define what a Lambda function **can access**, **resource-based policies** define **who** or **what** can invoke the Lambda function.

---

üìå **Use Cases:**

- ‚úÖ Allow **S3, API Gateway, or SNS** to invoke your Lambda
- ‚úÖ Enable **cross-account** access to Lambda

---

### üî• **Example: Allow S3 to Invoke a Lambda Function**

If an **S3 bucket** needs to **trigger a Lambda function**, you must add an **InvokeFunction permission** to the function‚Äôs **resource-based policy**.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "s3.amazonaws.com"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:myFunction",
      "Condition": {
        "StringEquals": {
          "AWS:SourceAccount": "123456789012"
        },
        "ArnLike": {
          "AWS:SourceArn": "arn:aws:s3:::my-bucket"
        }
      }
    }
  ]
}
```

> üìå **Breakdown:**
>
> - **Effect:** "Allow" ‚Üí Grants access
> - **Principal:** "s3.amazonaws.com" ‚Üí Allows S3 to invoke the function
> - **Action:** "lambda:InvokeFunction" ‚Üí Allows invocation
> - **Condition:** Ensures only **a specific S3 bucket** can trigger Lambda

---

### üîó **Example: Cross-Account Access to Lambda**

> If a Lambda function needs to be invoked from **another AWS account**, you need to add a **cross-account resource-based policy**.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::987654321098:root"
      },
      "Action": "lambda:InvokeFunction",
      "Resource": "arn:aws:lambda:us-west-2:123456789012:function:myFunction"
    }
  ]
}
```

> üìå **Breakdown:**
>
> - Allows **account `987654321098`** to invoke the function
> - Restricts the action to only `lambda:InvokeFunction`
>
> üí° **Best Practice:**
>
> - Use **specific IAM roles** instead of `root`
> - Restrict access to certain AWS regions or conditions

---

### üõ†Ô∏è **How to Add a Resource-Based Policy to Lambda**

#### **üñ±Ô∏è Using AWS Console**

- **1Ô∏è‚É£** Go to **AWS Lambda Console**
- **2Ô∏è‚É£** Select your **Lambda function**
- **3Ô∏è‚É£** Navigate to **Configuration** ‚Üí **Permissions**
- **4Ô∏è‚É£** Click **"Add Permissions"** ‚Üí **Resource-based policy**
- **5Ô∏è‚É£** Add a policy allowing the required AWS service or account

---

#### **üßë‚Äçüíª Using AWS CLI**

**1Ô∏è‚É£ Allow S3 to Invoke Lambda Function:**

```sh

--statement-id AllowS3Invoke \
--action "lambda:InvokeFunction" \
--principal s3.amazonaws.com \
--source-arn arn:aws:s3:::my-bucket \
--source-account 123456789012
```

> This allows **S3 bucket `my-bucket`** to invoke the Lambda function `myFunction`.

**2Ô∏è‚É£ Allow Another AWS Account to Invoke Lambda:**

```sh
aws lambda add-permission --function-name myFunction \
--statement-id AllowCrossAccountInvoke \
--action "lambda:InvokeFunction" \
--principal arn:aws:iam::987654321098:root
```

> This allows **AWS Account `987654321098`** to invoke `myFunction`.
